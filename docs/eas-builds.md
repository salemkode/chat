# Mobile EAS Builds

This repo now supports repeatable production builds for the Expo mobile app with EAS.

The standard path should be:

- keep app configuration in the repo
- keep production secrets in EAS environments, not copied by hand between worktrees
- use EAS Cloud for CI and release builds
- use `eas build --local` only for debugging or validating native issues on a developer machine

## What is standard now

- `eas.json` explicitly maps `development`, `preview`, and `production` profiles to matching EAS environments
- `eas.json` pins Bun `1.3.9` so cloud builds use the same package manager version we validated locally
- the repo postinstall script repairs two package-level issues that affected builds:
  - `react-native-worklets` bundle mode expects its `.worklets` directory to exist
  - `@expo/dom-webview` needs the current iOS patch applied during install

## What is intentionally local-only

- the React Native Foojay Gradle patch only runs when `EAS_BUILD_RUNNER=local-build-plugin`
- Homebrew-specific `pod` and `fastlane` fixes that were used during local debugging are not part of this repo and are not required for EAS Cloud

That split is important: cloud builds should rely only on committed files plus EAS configuration, while local builds may still depend on the developer machine having Android Studio, Xcode, CocoaPods, and Fastlane installed correctly.

## Environment variables

For cloud builds, use EAS environments instead of copying `.env` files around by hand.

Expo documents that EAS environments are the supported way to provide build-time variables to cloud jobs, and that local `.env` files are not available on remote builders. See:

- [Environment variables in EAS](https://docs.expo.dev/eas/environment-variables/)
- [Using environment variables with EAS Build](https://docs.expo.dev/eas/environment-variables/usage/)

Recommended workflow:

```bash
eas env:pull --environment development --path apps/mobile/.env.local
eas env:pull --environment production --path apps/mobile/.env.local
```

For production PR validation, prefer making sure the `production` EAS environment is complete rather than copying secrets from another checkout.

## Local vs cloud

Expo documents that local builds are mainly for debugging and differ from cloud builds in a few ways, including toolchain responsibility and secret handling:

- [Run EAS Build locally with `--local`](https://docs.expo.dev/build-reference/local-builds/)

Practical guidance for this repo:

- use EAS Cloud as the source of truth for release readiness
- use local builds to debug native failures before or after a cloud failure
- avoid adding machine-specific paths or binaries to repo scripts
- keep the native iOS project generated by Expo/EAS instead of committing it into the branch

## Release commands

Typical production builds:

```bash
eas build --platform android --profile production
eas build --platform ios --profile production
```

Local debugging builds:

```bash
eas build --platform android --profile production --local
eas build --platform ios --profile production --local
```
